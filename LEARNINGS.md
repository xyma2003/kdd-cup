# KDD Cup 2026 DataAgent-Bench — 竞赛复盘与学习总结

## 竞赛概述

**比赛**：KDD Cup 2026 DataAgent-Bench  
**任务**：给定自然语言问题 + 异构数据（CSV/JSON/SQLite/文本），Agent 自主探索数据并输出答案表格  
**架构**：放弃 ReAct 基线，从零设计基于规则分类器 + 专用 Solver 的自定义架构  
**最终得分**：~32 / 50（公开 demo 集，Qwen3.5-35B）

---

## 一、架构设计决策

### 1. 放弃 ReAct，选择专用 Solver

ReAct 是官方基线，30步探索循环，模型自主决定调用哪个工具。我们选择完全放弃它，改用：

```
分类器（纯规则）→ 路由到专用 Solver
├── TYPE_SQL    → sql_solver（DuckDB 跨源查询）
├── TYPE_PANDAS → pandas_solver（DataFrame 操作）
├── TYPE_HYBRID → structured_solver（文档提取 + 代码生成）
└── TYPE_DOC    → structured_solver
```

**为什么这样做**：ReAct 每步都要 LLM 决策，30步 × thinking mode = 极慢且不可控。专用 Solver 把问题结构化，每类题只做该做的事，更快更准。

**教训**：专用架构在已知题型上效率高，但对未见题型的泛化能力弱于 ReAct。trade-off 要根据题型分布决定。

### 2. 分类器的设计哲学

分类器零 LLM 调用，纯规则：有 `.db` → SQL，有 `.md`/`.txt` doc → HYBRID，否则 → PANDAS。

**教训**：分类器错误是系统性失败的根因之一。`knowledge.md` 被误识别为 doc 文件曾导致题目路由错误。规则分类器需要细心处理边界情况。

## 二、Thinking Mode 的使用经验

### 开启方式
SiliconFlow 的 Qwen3.5 把答案放在 `reasoning_content` 而非 `content`，需要特殊处理：

```python
if is_qwen3 and not content.strip():
    content = _extract_final_answer_from_thinking(reasoning_content)
```

### 收益与代价
- **收益**：复杂题（COUNT DISTINCT、JSON嵌套、跨表逻辑）准确率显著提升，v4→v5 净增 +5分
- **代价**：每次调用 30-180s，多轮调用容易超时；temperature=0 不能完全消除随机性，同一道题两次结果可能不同

### 关键教训
**Thinking mode 引入了不可忽视的随机性**。同一套代码跑两次，分数可能差 2-3 分。这意味着：
- 本地评测分数不等于真实能力
- 需要多次跑取均值才能判断优化是否有效
- "这次分数高"可能只是运气

## 三、Self-Verifier 的双刃剑效应

我们加入了 SQL 执行后的语义复查：LLM 检查结果形状是否符合问题语义，不对则带诊断重新生成 SQL。

**正收益**：修复了列名含聚合函数、多余行等问题（+task_305、task_330）  
**负收益**：task_200 被 verify 把正确的单行结果判成"应该多行"，改坏了（-1.0）

**本质问题**：用 LLM 验证 LLM 的输出，误判概率不为零。Verify 能抓住的错误类型有限（明显行数异常），但也会把正确结果判错。

**教训**：Self-verifier 应该非常保守，只在结果有明显异常时触发（如空结果、行数爆炸）。对于聚合查询（COUNT/SUM/AVG 无 GROUP BY），结果必然是1行，永远不应该被质疑。

## 四、Prompt 工程的实际效果

### 有效的规则
- **Tied rows**：`WHERE cost = (SELECT MIN(cost) FROM ...)` 而不是 `LIMIT 1`，确保返回所有并列行
- **列裁剪**：`Tally/list X of Y → return only X`，不返回 Y
- **精度保留**：不 ROUND，保持完整浮点精度（举办方评分会归一化到2位，但过程中精度损失影响中间计算）

### 无效或有害的规则
- 过多规则会让 LLM 困惑，甚至在原本正确的情况下"修正"出错误答案
- 规则之间可能产生冲突，导致不可预期的行为

**教训**：Prompt 规则要少而精，每条规则都应该有明确的 bad case 驱动。加规则前想清楚：这条规则在什么情况下会误伤？

## 五、文档处理架构的演进

### v1-v4：只支持 ID 映射型文档
假设所有 HYBRID 题的 doc 文件都是"提供 rec_id → name 映射"，用正则提取后 JOIN 结构化数据。

### v5：BM25 大文档筛选
大文档（>12000字符）先用 BM25 关键词打分筛选相关段落，解决了 LLM context 超长和超时问题。

### v6 尝试：直接数据型文档
当 ID 映射提取失败时，把文档原文传给 LLM，让它用 Python regex 直接提取数值。思路对，但实现时发现：
1. 正则提取器会把散文片段误认为 ID 映射（需要噪声过滤）
2. 某些题的答案根本不在任何文件里（数据缺失，无解）

**教训**：文档用途分类（ID映射型 vs 直接数据型）是个好方向，但需要更精准的判断标准。噪声过滤的阈值设置需要在多个样本上验证，不能只看单个 case。

## 六、实验方法论的教训

### 改动要原子化
每次只改一个变量，才能知道分数变化的真正原因。我们有几次同时改多处，导致退步时无法定位根因。

### 区分"代码问题"和"随机波动"
thinking mode 下同一道题两次结果可能不同。判断一个优化是否有效，需要：
- 至少跑2次全量对比
- 关注**系统性**变化（多道题同方向变化），而不是单道题的得失

### 及时验证提交链路
我们最严重的错误：`use_xyma=False` 默认值导致举办方评测走了原始 ReAct，自己写的所有代码完全没有被执行，得了 0 分。

**教训**：在第一次提交前，应该 end-to-end 验证整个提交链路——从 `entrypoint.sh` 到最终 `prediction.csv` 输出，每个环节都要确认。不要只测局部功能。

## 七、技术栈总结

| 组件 | 技术选择 | 原因 |
|------|---------|------|
| SQL 执行 | DuckDB | 可以同时查 SQLite + CSV + JSON，一条 SQL 跨源 |
| 文档筛选 | BM25（自实现） | 无需外部依赖，关键词匹配足够 |
| LLM 调用 | OpenAI SDK + SiliconFlow | OpenAI 兼容接口，换模型只改配置 |
| 容错解析 | 5层 JSON fallback | LLM 输出格式不规范是常态，必须有多层兜底 |
| 任务路由 | 纯规则分类器 | 零 LLM 开销，快速稳定 |

---

## 八、如果重来会怎么做

1. **第一天就验证提交链路**，不要等到提交了才发现 bug
2. **建立严格的实验记录**：每次改动前记录 baseline，改动后跑2次对比
3. **thinking mode 慎用**：带来准确率提升，但也带来随机性和超时风险，需要权衡
4. **不要针对单个 case 改代码**：每次改动都要问"这个改动在未见题上也有效吗"
5. **更早做架构层面的思考**：我们花了太多时间在 prompt 规则上，而 HYBRID solver 的架构问题才是真正的瓶颈
